Seamless Registration Webcasts

Seamless Registration is a service for pushing registrant data to Elite. This service supports both the GET and POST HTTP methods.

 

Best Practices

The simplest approach is to redirect users to ON24 once they’ve finished filling out a registration page on a third party site. Users will see an ON24 hosted “Thank you for registering, this event will start on….” Lobby page, and an ON24 cookie is saved that serves as the persistent storage to identify the user. With the cookie present, clicking on the Audience URL will bypass the ON24 login/registration page. You also have the option of having ON24 automatically redirect to a generic third party hosted “thank you for registering” page instead, by passing a URL in the optional “lobby” parameter.

 

To see an example of a Seamless Registration page, click here.

 

Note: Be sure the Enable Registration and Enable Login Only Option checkboxes are checked to use Seamless Registration. You can find the checkboxes under "Options" on the webinar's registration page.

 

Parameter Notes

Below you’ll find the fields that can be passed to the ON24 registration database. Please note the following:

  • The parameter names are case-sensitive.
  • You may have changed the display label of the field, but the underlying parameter name is always retained and should be referenced when passing registration data to ON24.
  • Associate the fields with the correct parameter name (e.g. pass the attendee’s first name in the "firstname" field rather than in some other field).
  • If the data you want to store is not in one of the named demographic fields, use the custom fields std1-std10.
  • Any values sent which exceed the designated character limit will be truncated.
  • Records containing the email address of a registrant who is already in the database for this webcast will be updated with the latest values being sent.
  • The webcast’s unique eventid and key parameters must be included regardless of the method you use to send the Registrant data. These values are contained within the Audience URL: https://event.on24.com/wcc/r/eventid/key

HTTP GET Method

The GET method relies on JavaScript redirection, which requires a web browser. This method will fail if used for server-to-server requests. Note the below additional information:

  • If any of the parameters include the SPACE character, it must be defined using URL encoding i.e. " ", (eg, &country=United States). Some special characters embedded within the field values, like "=", "&" etc., should also always be encoded.
  • It is recommended that you add the deletecookie=true parameter to avoid issues with users regsitering for the same event on one machine.
  • Example URL: https://event.on24.com/interface/registration/autoreg/index.html?eventid=XXXXX&key=XXXXXXXXXXXXXXXXXXX&deletecookie=true&firstname=John&lastname=Smith&email=john.smith@abc.com&company=ABC
  • If you are having trouble with the standard HTTPS GET URL, you can try https://event.on24.com/utilApp/r instead.

 

HTTP POST Method

  • When in a batch mode or registering multiple users, you must also send the additional deletecookie=true parameter.
  • UTF input isn’t necessary if sending UTF-8, which is the default. If you’re sending characters using a different encoding, change the hidden value to the proper character set. For example, ISO-8859-1.
  • Example request:

<form name="myform" action="https://event.on24.com/utilApp/r" method="POST">

<input type="hidden" name="eventid" value="XXXXX">

<input type="hidden" name="key" value="XXXXX">

<input type="text" name="firstname" value="john">

<input type="text" name="lastname" value="smith">

<input type="text" name="email" value="john.smith@abc.com">

<input type="text" name="company" value="ABC, Inc.">

</form>

 

Additional Notes

  • The Seamless Registration process will truncate any values that are sent with a greater number of characters than the designated limit.
  • If a registrant record with the same email value of a registrant already exists in the database, the auto-registration script will update the record with the latest values being sent to the script. If the email value does not match an existing registrant record, a new record will be added to the database.
  • Cannot guarantee results if fields with predefined business rules are used for other than defined purposes.
  • When using HTTP GET, if any of the parameters includes the SPACE character, it must be defined using URL encoding i.e. " ". For example, &country=United States, this equates to "United States". You do not need to encode the parameter values for POST.
  • When using HTTP GET, some characters embedded within the field values, such as "=", "&" etc., present the possibility of being misunderstood within URLs for various reasons. These characters should also always be encoded. You do not need to encode the parameter values for POST.
  • The POST URL: https://event.on24.com/utilApp/r may also be used for the GET if you are having trouble with the Standard HTTP GET URL.
  • If you use multiple registration pages for the same webcast, you can include a Partner Referral code to track the individual campaigns.
